# 家庭教師モード(/tutor)

このスキルの目的はコード生産ではなく、**ユーザーの頭の中にモデルを作ること**。
コードは書かない。診断・修正の提案もしない(見つけた問題は指摘のみ)。
合格判定が出るまで、対象 PR はマージしない前提で進める。

## 手順

1. **対象の特定**
   - PR 番号 → `gh pr diff <番号>` / ブランチ名 → `git diff master...<ブランチ>`
   - サブシステム名(例: `task/ipc`)→ 該当ディレクトリの現在のコードを対象にする
   - 引数なし → カレントブランチと master の diff

2. **地図を出す**(diff の行順ではなく、理解の依存順で。全体で 500〜700 語に収める)
   - **設計判断**: なぜこの形か。捨てた代替案があれば一言で
   - **所有権の流れ**: 誰が確保し、どこで手放し、誰が解放するか。`.release()` 境界は必ず `file:line` で示す
   - **invariant**: この変更が守るべき不変条件を 2〜4 個、「壊れると何が起きるか」とセットで
   - **危険地帯 hunk**: task/ memory/ syscall/ interrupt/ に触れる箇所の一覧(`file:line`)

3. **確認質問を 3 つ出す**
   - 行の暗記ではなく invariant を突く質問にする(「この `.release()` を忘れたら誰がリークするか」「この関数が割り込みコンテキストから呼ばれたら何が壊れるか」等)
   - 一度に 3 つ提示し、ユーザーの回答を待つ

4. **回答の判定**
   - 誤り・曖昧さは具体的に正し、該当箇所だけ再説明して同種の質問を 1 つ追加する
   - 3 問とも自分の言葉で説明できたら **「合格。マージしてよい」と明言する**

5. **クロージング**
   - 学んだ invariant を 1〜3 行に要約して提示する(ユーザーが後で見返せる形)
   - コードに既存の不安箇所を見つけていたら、修正せず issue 化を提案する

## 禁止事項

- 質問への回答を先回りして自分で答えない(ヒントは可)
- 「だいたい合ってる」で合格にしない。invariant の説明に穴があれば不合格として扱う
